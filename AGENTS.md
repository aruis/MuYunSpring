# Agent Guide

本仓库是 `MuYunSpring`。核心路线、里程碑和架构边界以 `docs/` 为准，Agent 不应在任务中重新发明方向。

## Read First

非 trivial 任务前按顺序阅读：

1. `README.md`
2. `docs/DEVELOPMENT_PRINCIPLES.md`
3. `docs/roadmap/MILESTONES.md`
4. `docs/architecture/DYNAMIC_STATIC_UNIFIED_CORE.md`
5. `docs/architecture/NAMING_AND_BOUNDARIES.md`

只详细展开当前里程碑。后续里程碑不要提前设计成完整方案。

## Execution

1. 主 agent 负责目标理解、任务拆解、边界判断、调度执行、质量复核和最终交付。
2. 小任务可直接处理；任务较大、跨模块、需要调研、并行实现或独立 review 时，鼓励交给 subagent。
3. subagent 任务必须有清晰范围和产出要求。主 agent 负责整合意见并做最终判断。
4. 任务拆解围绕完整业务目标、用户场景或工程能力，不围绕方便提交的零散改动。
5. 新增想法先归类为当前要做、后续预留、明确不做或风险记录；会扩大范围但符合中长期利益的方案，应先与用户确认。

## Engineering Rules

1. 不迷信“最小改动”。当问题影响基础边界、公共模型、长期维护或后续开发时，应整理到合理状态。
2. 分层、模块和接口暴露面要着眼于中长期利益；确有必要时，允许通过重构修正边界。
3. 优先沿用项目已有架构、命名、工具和测试风格。引入新做法或外部依赖前，必须说明它解决的真实问题、替代方案和维护风险，并与用户确认。
4. 拆分模块、类、函数或文件必须带来更清晰的职责、更低理解成本或更好的变化隔离。
5. 普通业务接入应保持轻量，不要把业务模块推向低层基础设施细节。
6. 推进能力或业务块后，要检查它是否减少重复代码、降低接入成本。

## Repository Boundaries

1. 默认只修改当前仓库。遇到同名目录、实验目录或相邻项目时，先确认当前任务是否真的需要跨仓库。
2. 本机绝对路径、私有目录结构和相邻仓库位置只记录在本地 ignored 文件中，不写入公开文档。
3. 只有底座确实需要调整时，才跨仓库修改，并说明原因、影响和验证方式。

## Tests And Docs

1. 默认验证命令是 `./gradlew test`。
2. 对外可见行为、关键边界和已确认契约必须有测试或明确说明支撑；行为偏差修正后应补回归测试。
3. 涉及真实数据访问、字段解析、分页排序、建表等基础边界时，应补真实路径 contract 测试，不能长期只依赖 fake。
4. 文档记录稳定目标、关键边界、公共契约、重要决策、技术债和交接信息，不记录探索过程、复核过程或执行流水。
5. 能由测试描述的细节，不写成长篇文档。

## Review And Delivery

1. Review 先看方向和边界，再看行为正确性，最后看代码风格。
2. 如果发现方向或边界偏差，要及时汇报。
3. Review 发现的问题不只处理 P0；P1、P2 中工作量不大且符合项目利益的问题，也鼓励一并修复。
4. 对平台中长期建设有重大影响的方案，应引入 subagent 做独立 review 或方案碰撞，并向用户汇报取舍。
5. 提交前默认运行 `./gradlew test`；commit message 使用 Conventional Commits 类型加中文描述，例如 `feat: 夯实核心能力契约`；未被明确要求时不要自动 push。

## Reporting

汇报进度时说明完成了什么能力、解决了什么问题、验证了什么结果、剩余什么风险，并给出代码变更数量：

```text
生产代码：+N / -N
测试代码：+N / -N
代码合计：+N / -N
含文档总计：+N / -N
```

## Do Not

1. 不要引入与当前路线无关的框架或平台机制。
2. 不要提前实现当前里程碑之外的能力，除非底座需要先明确契约。
3. 不要创建只有占位意义的 Gradle 子项目、包或类。
4. 不要把讨论过程、探索材料或过渡语境写入正式文档。
