# Agent Guide

本仓库是 `MuYunSpring`。核心路线、里程碑和架构边界以 `docs/` 为准，Agent 不应在任务中重新发明方向。

## Read First

非 trivial 任务前按顺序阅读：

1. `README.md`
2. `docs/DEVELOPMENT_PRINCIPLES.md`
3. `docs/roadmap/MILESTONES.md`
4. `docs/architecture/DYNAMIC_STATIC_UNIFIED_CORE.md`
5. `docs/architecture/NAMING_AND_BOUNDARIES.md`

只详细展开当前里程碑。后续里程碑不要提前设计成完整方案。

## Execution

1. 主 agent 负责目标理解、任务拆解、边界判断、调度执行、质量复核和最终交付。
2. 默认由主 agent 推进实现、验证和交付。subagent 优先承担独立 review、专题调研和方案碰撞，不默认作为开发主力。
3. 只有任务边界清晰、写入范围可隔离、不会外包核心设计判断时，才把实现工作交给 worker subagent。
4. 平台基础设施、公共能力、跨模块边界、动静一体链路、命名边界和结构美学，应按风险安排一层或多层 subagent review。
5. subagent 任务必须有清晰范围和产出要求。主 agent 负责整合意见、独立复核并做最终判断；subagent 意见不是命令。
6. 一系列业务线或能力线相近的连续 review，优先复用同一个 subagent，以保留上下文和判断口径；只有任务明显换域、agent 卡住或需要独立第二意见时才另开。
7. 任务拆解围绕完整业务目标、用户场景或工程能力，不围绕方便提交的零散改动。
8. 新增想法先归类为当前要做、后续预留、明确不做或风险记录；会扩大范围但符合中长期利益的方案，应先与用户确认。

## Platform Principles

1. 核心路线是“动静一体”：静态 Java 模型和动态元数据模型应复用同一套平台能力，而不是形成两套平行实现。
2. 能力层先夯实，再铺业务线。CRUD、数据访问、模型基类、动态建表、租户、软删、事务、缓存、树、排序、引用、子表等基础能力要做到功能完备、封装良好、开箱即用。
3. 静态链路优雅优先。动态链路主要由平台维护；静态链路会被业务服务大量编写。两者取舍冲突时，优先保证静态模型和静态 service 直观、省代码、低心智负担。
4. 动态侧按元数据声明能力，运行态按能力组合执行；静态侧通过模型契约、注解、基类和能力接口接入平台。不要把低代码路线退化成业务硬编码。
5. 能力启用后的标准字段名和语义要稳定。树、排序、软删、启停、租户、引用等能力在动静两侧应尽量使用一致字段和一致行为。
6. 平台推荐使用稳定门面 API。内部运行态已有能力但门面没有暴露时，业务可用性仍不完整，应补齐门面和契约测试。
7. 借鉴成熟设计，但不复制历史包袱。模型命名、包边界、接口暴露面可以按当前项目的理想形态重建。

## Engineering Rules

1. 不迷信“最小改动”。当问题影响基础边界、公共模型、长期维护或后续开发时，应整理到合理状态。
2. 分层、模块和接口暴露面要着眼于中长期利益；确有必要时，允许通过重构修正边界。
3. 优先沿用项目已有架构、命名、工具和测试风格。引入新做法或外部依赖前，必须说明它解决的真实问题、替代方案和维护风险，并与用户确认。
4. 拆分模块、类、函数或文件必须带来更清晰的职责、更低理解成本或更好的变化隔离；不为了拆而拆，也不为了“少改”保留坏结构。
5. 普通业务接入应保持轻量，不要把业务模块推向低层基础设施细节。
6. 推进能力或业务块后，要检查它是否减少重复代码、降低接入成本。
7. 压缩代码量的前提是优雅、可读、易理解、易使用；不要为了省代码而隐藏复杂度或牺牲业务直觉。
8. 公共能力要比当前调用点多考虑几步，至少保证命名、接口、错误语义、集成方式和测试方式能支撑中期演进。
9. 项目仍处于开发阶段，不为不合理的过渡设计背兼容包袱。发现边界、命名、模型或暴露面不理想时，可以重构到合理状态，并用测试锁住新契约。
10. 新增框架、平台机制、里程碑外能力、Gradle 子项目、包或类时，先判断是否符合平台中长期利益；若确有必要，应说明收益、风险、边界和验证方式。

## Repository Boundaries

1. 默认只修改当前仓库。遇到同名目录、实验目录或相邻项目时，先确认当前任务是否真的需要跨仓库。
2. 本机绝对路径、私有目录结构和相邻仓库位置只记录在本地 ignored 文件中，不写入公开文档。
3. 只有底座确实需要调整时，才跨仓库修改，并说明原因、影响和验证方式。

## Tests And Docs

1. 默认验证命令是 `./gradlew test`。
2. 对外可见行为、关键边界和已确认契约必须有测试或明确说明支撑；行为偏差修正后应补回归测试。
3. 涉及真实数据访问、字段解析、分页排序、建表等基础边界时，应补真实路径 contract 测试，不能长期只依赖 fake。
4. 文档记录稳定目标、关键边界、公共契约、重要决策、技术债和交接信息；不要记录讨论过程、探索材料、复核过程、执行流水或过渡语境。
5. 能由测试描述的细节，不写成长篇文档。
6. 文档边界应随阶段推进及时回收。不要让“不要、禁止、暂不做”类文字长期阻碍业务开发；如果边界已经过期，应主动精简或调整。

## Review And Delivery

1. Review 先看方向和边界，再看行为正确性，最后看代码风格。
2. 如果发现方向或边界偏差，要及时汇报。
3. Review 发现的问题不只处理 P0；P1、P2 中工作量不大且符合项目利益的问题，也鼓励一并修复。
4. 对平台中长期建设有重大影响的方案，应引入 subagent 做独立 review 或方案碰撞，并向用户汇报取舍。
5. 阶段性进展后应及时 review、修复、验证并提交；未被明确要求时不要自动 push。
6. 提交前默认运行 `./gradlew test`；commit message 使用 Conventional Commits 类型加中文描述，例如 `feat: 夯实核心能力契约`。

## Reporting

汇报进度时说明完成了什么能力、解决了什么问题、验证了什么结果、剩余什么风险，并给出代码变更数量：

```text
生产代码：+N / -N
测试代码：+N / -N
代码合计：+N / -N
含文档总计：+N / -N
```
