# 平台配置专题

## 能力定位

平台配置专题覆盖 MuYunSpring 自身配置态的基础对象和发布链路，目标是让应用、模块、元数据、字段、数据字典和菜单等配置按平台能力维护，并能发布为动态运行态。

这条链路服务于“动静一体”：动态模块通过元数据声明能力，发布后进入与静态模型一致的数据访问、能力语义、生命周期、权限、审计、租户和事务边界。专题文档只梳理配置对象和业务线索，不展开 Java 底层实现。

Web 维护面按“独立配置根 + 模块聚合子资源”组织：应用、模块、元数据保留平台模块 alias 根路径；模块动作、模块-元数据关系、模块字段配置从 `/platform.module/{moduleAlias}` 进入。发布、导入导出、回滚和健康检查仍归属配置治理，不混入普通维护接口。

## 主要对象

| 对象 | 作用 | 关键边界 |
| --- | --- | --- |
| 应用 | 平台业务空间的上层归属 | `applicationAlias` 全局唯一，使用单段小写标识 |
| 模块 | 运行时业务边界 | 统一使用 `moduleAlias`，格式为 `<applicationAlias>.<moduleName>`；不使用 `moduleId` 表达同一件事 |
| 元数据 | 动态实体的配置态定义 | `Metadata.id` 是平台稳定 ID；`metadataAlias` 在应用内唯一；物理定位使用 `schemaName + tableName` |
| 字段 | 元数据的字段事实和轻量行为 | 表达字段名、列名、类型、约束、字典绑定、引用、保护和公式等运行态必须知道的信息 |
| 字段类型与 UI 类型 | 字段配置的基础目录 | 字段类型定义运行态形态和查询能力；UI 类型定义页面控件的可选目录 |
| 模块-元数据关系 | 模块与主/子元数据的绑定关系 | 一个模块最多一个主元数据；动态模块必须有主元数据；关系可表达父、子、孙结构 |
| 数据字典 | 字段选项和值域约束 | 类目 alias 在应用内唯一；项目 code 在类目内唯一；业务数据保存字典项目 `code` |
| 菜单方案与菜单 | 基础入口组织 | 菜单不归属应用；一个菜单只属于一个方案；模块可被多个菜单节点挂载 |

## 核心业务链路

1. 创建应用，确定 `applicationAlias`。
2. 创建模块，确定稳定 `moduleAlias`，并声明静态或动态接入方式。
3. 为动态模块维护元数据、字段、关系和字典绑定。
4. 通过模块-元数据关系绑定主元数据和子元数据。
5. 配置菜单方案和菜单节点，将模块挂载为运行入口。
6. 发布配置，编译为动态 `ModuleDefinition`、`EntityDefinition`、`FieldDefinition` 等运行态定义。
7. 发布链路执行建表或 schema 更新，并刷新动态运行态。
8. 运行态通过动态记录入口按 `moduleAlias + metadataAlias` 执行描述读取、CRUD、查询和引用等能力。

最小闭环是：

```text
应用 -> 模块 -> 元数据 -> 字段/关系/菜单/字典 -> 发布 -> DynamicRecordService -> CRUD/query
```

## 边界说明

1. 配置专题聚焦应用、模块、元数据、字段、数据字典、菜单基础配置和发布到运行态。
2. 静态模块可以没有元数据；动态模块必须有一个主元数据。
3. 字段定义只承载运行态必须知道的轻量字段事实和行为；引用过滤、引用回填、字段保护和公式规则属于配置态字段行为，不扩展成独立业务 Controller。
4. 菜单模型只表达入口树和模块挂载；权限剪枝属于权限专题，不能反向污染菜单配置。
5. 运行态 URL、权限、审计和 OpenAPI 统一围绕 `moduleAlias`，不暴露内部实体定位作为业务入口。
6. 配置发布是配置态到运行态的边界，不应在普通业务保存动作中隐式触发表结构变更。
7. 本专题不复盘推进过程，不记录测试流水；稳定契约优先由代码和 contract 测试支撑。

## 关联专题

- 动态模块运行态：描述读取、动态 CRUD、查询、引用解析、动作执行、OpenAPI。
- 权限与身份：动作授权、数据权限、菜单剪枝、当前用户入口。
- 页面交付：菜单入口 bootstrap、列表/表单 UI 配置、个人偏好。
- 配置治理：配置包、健康检查、版本发布、发布回滚、迁移导入导出。
- 平台执行底座：规则、公式、动作上下文、运行事件和审计。
