# 身份与权限专题概览

身份与权限专题把租户、组织、部门、用户、角色、动作授权、数据权限、菜单剪枝和当前用户上下文串成同一条后端治理链路，为静态模块和动态模块提供一致的身份来源、授权判断和数据范围约束。

本文只梳理平台能力和接入线索，不展开 Java 底层设计。实现细节以 `muyun-iam` 领域服务、`muyun-boot` Web 入口和相关 contract 测试为准。

## 能力定位

身份与权限不是独立于平台能力之外的业务模块，而是动作执行、动态 Web、静态 Web、菜单返回和审计上下文的共同前置能力。

核心定位：

1. 租户是平台数据隔离的身份根，`tenantId` 仍是基础字段和默认 Ability 作用域。
2. 当前用户上下文负责把登录用户、租户、组织和系统态操作者带入请求链路。
3. 动作权限约束「能不能执行某个模块动作」。
4. 数据权限约束「能在该动作下看到或操作哪些记录」，不替代租户基线隔离。
5. 菜单权限只做返回树剪枝，不改变菜单模型、菜单方案或菜单挂载关系。
6. 静态模块和动态模块都按 `moduleAlias + actionCode` 进入同一套权限口径。

## 主要对象

| 对象 | 标识 | 说明 |
| --- | --- | --- |
| 租户 | `iam.tenant` | 平台身份根。租户 alias 当前等同于租户 ID；租户自身由系统态维护，支持启停和排序。 |
| 组织机构 | `iam.organization` | 租户内机构树，表达公司、分支机构、事业部等组织主体；支持树、排序、启停，机构编码在租户内唯一。 |
| 部门 | `iam.department` | 归属某个组织机构的部门树；支持树、排序、启停，部门编码在同一机构内唯一。 |
| 职员 | `iam.employee` | 租户内人员主数据，归属机构和部门，承载员工号、姓名、联系方式、启停状态和排序。 |
| 职员账号 | 子资源 | 职员与登录账号的绑定关系，通过职员入口维护；当前一个账号只绑定一个职员。 |
| 岗位 | `iam.position` | 租户内岗位标准项，不强制绑定部门，用于维护可复用岗位定义。 |
| 职员任岗 | 子资源 | 职员在某个机构、部门担任某个岗位的关系事实，通过职员入口维护。 |
| 用户 | `iam.user` | 租户内登录账号，包含用户名、联系方式、机构、启停状态和密码哈希；用户名在租户内唯一。 |
| 角色 | `iam.role` | 租户内权限包，包含标准角色、岗位模板角色、系统角色、角色组和数据权限通配角色；角色声明自身允许授给账号、职员或具体任岗。 |
| 角色授权实例 | 授权事实 | 统一表达角色授给账号、职员或具体任岗的事实，是多账号、职员授权和任岗授权的共同承载。 |
| 角色动作授权 | 授权事实 | 角色到 `moduleAlias + actionCode` 的授权事实，包含数据权限策略和租户范围策略。 |
| 当前用户 | `CurrentUser` | 请求内身份快照，包含 `userId`、`username`、`tenantId`、`organizationId` 和 `system`。 |

## 核心业务链路

### 登录与上下文

1. 用户调用登录入口提交 `tenantId + username + password`。
2. 登录服务在租户上下文内校验用户状态和密码，生成 Bearer token。
3. 后续请求由 Bearer token 解析当前用户。
4. `CurrentUserWebFilter` 将当前用户写入 `CurrentUserContext`；普通租户用户同步写入 `TenantContext`，系统用户进入系统态租户上下文。

### 组织、部门、职员与用户

1. 租户由系统态维护，是用户、组织、部门、角色和菜单方案等租户内对象的上层边界。
2. 组织机构在租户下形成树，表达公司、分支机构、事业部等主体。
3. 部门归属组织机构，并在机构内形成部门树；部门不跨机构复用。
4. 职员归属一个主机构和一个主部门，用于承载清晰的人事身份主数据。
5. 岗位是租户内可复用标准项，不直接表达某个部门下的任职关系。
6. 职员任岗关系表达某个职员在某个机构、部门担任某个岗位；维护入口归入职员体系。
7. 用户账号归属租户，可绑定机构；用户自身也是可参与数据权限判断的数据范围对象。
8. 职员账号绑定表达登录账号对应的职员身份；当前一个账号只绑定一个职员，不做一个账号多职员身份切换。

### 角色与授权

1. 标准角色承载普通动作授权。
2. 岗位模板角色用于后续授给具体任岗，让“本机构”等相对数据范围按任岗上下文解释；部门范围策略后续随任岗授权上下文补齐。
3. 角色组聚合标准角色，不直接混入任意角色类型。
4. 数据权限通配角色用于补充数据范围；一个用户最多绑定一个。
5. 角色本身不绑定机构、部门或任岗；这些归属由授权实例提供，避免按组织复制角色。
6. 授权事实按 `moduleAlias + actionCode` 落表；配置动作和标准动作通过授权矩阵统一回显。
7. 动作继承会归并到最终权限动作，避免同一入口出现多套授权口径。

### 动作权限与数据权限

1. Web 入口和动作执行入口通过动作上下文声明当前模块、动作、动作级别和是否需要数据权限。
2. 动作授权先判断当前用户是否具备该模块动作。
3. 需要数据权限的动作继续解析数据范围策略。
4. 查询类动作将数据范围合入查询条件；记录类动作要求目标记录落在授权范围内。
5. 动态模块标准 CRUD、动态动作列表、记录动作可用性、descriptor 和 OpenAPI 都接入同一动作授权结果。

当前数据范围策略包括：

| 策略 | 含义 |
| --- | --- |
| `none` | 无数据权限 |
| `all` | 全部数据 |
| `owner` | 本人负责 |
| `assignee` | 负责人 |
| `member` | 相关人 |
| `organization` | 本机构 |
| `organizationAndChildren` | 本机构及下级 |
| `wildcard` | 通配数据权限 |
| `referenceDependency` | 引用依赖 |
| `custom` | 保留字段，当前授权入口拒绝开放 |

租户范围策略包括当前租户和全部租户。它只描述授权事实的租户适用范围，不取消平台基础租户隔离的默认语义。

相对数据范围需要结合授权上下文解释。账号直绑角色缺少稳定机构、部门上下文，适合承载全局、本人或显式账号级权限；职员和任岗来源的角色可以分别按职员主机构、具体任岗机构解释“本机构”等已实现范围。后续引入部门数据范围后，再按职员主部门或具体任岗部门解释“本部门”。当前账号权限计算已消费账号类型的角色授权实例；职员和任岗授权先作为有效授权聚合的事实来源，后续再进入统一 effective grant 计算。

### 菜单剪枝

菜单模型仍归平台菜单专题维护。权限专题只在返回当前用户菜单树时做可见性剪枝：

1. 后端根据当前用户推理菜单方案，前端不传 `menuSchemeAlias`。
2. 菜单节点挂载模块后，按模块 `menu` 动作判断是否可见。
3. 剪枝只影响本次返回结果，不修改菜单节点、菜单方案或模块挂载。

## 组织人员主数据

身份权限从账号权限延伸到组织人员主数据，但领域命名继续沿用当前平台语义，不为机构另造新名词。

### 组织与部门

`Organization` 继续表达机构语义，例如公司、分支机构、事业部等相对稳定的组织主体。机构可以形成树，用于表达集团、公司、分支机构之间的管理归属。

`Department` 表达部门模型。部门必须归属某个 `Organization`，并在机构内形成部门树。部门不跨机构复用；跨机构协作通过人员任职、兼职或后续人员组表达。

组织和部门拆开后，身份上下文和数据权限需要区分两类范围：

1. 机构范围：当前机构、机构及下级。
2. 部门范围：当前部门、部门及下级。

### 职员与用户

`Employee` 表达职员主数据。职员承载员工号、姓名、联系方式、主机构和主部门等人事身份，不等同于登录账号。员工号在同一机构内唯一，职员所在部门必须归属同一机构。

职员任岗关系表达职员的岗位事实。主岗必须匹配职员主机构和主部门；兼岗可以指向其他有效机构和部门，但当前只作为人事关系事实，不参与权限推导、工作流参与人解析或自动同步职员主部门。

`UserAccount` 继续表达登录账号。账号与职员绑定作为职员子资源维护，是后续从登录用户聚合职员授权、任岗授权和兼职上下文的前置事实。当前只支持一个账号绑定一个职员，不做一个账号多职员身份切换；账号与职员生命周期联动后续单独收口。

### 岗位与职务

`Position` 表示岗位标准项，例如销售经理岗、财务审核岗。岗位标准项不强制关联部门；某个部门下实际可任职岗位通过职员任岗等关系事实表达。

`JobTitle` 可作为轻量职务/职称模型，表达部门经理、总监、高级工程师等称谓或职责级别。暂不单独建设“职位”模型，避免和岗位、职务混用；如果后续出现岗位族、职级、招聘职位等明确场景，再独立扩展。

跨机构、跨部门兼职以及兼职场景下的数据权限暂不进入当前建设范围，后续按独立专题讨论。

## 边界说明

1. 本专题不把身份权限设计成独立低代码 DSL；动态模块和静态模块都回到平台动作、数据范围和菜单剪枝能力。
2. 数据权限不和 `tenantId` 基线隔离混淆；租户过滤仍是平台默认作用域。
3. 菜单权限不改变菜单配置事实，只裁剪当前用户可见树。
4. 平台动作审计只记录必要身份和权限判定上下文，不替代工作流、导入导出等专题流水。
5. 当前不展开完整管理端 UI、自定义数据权限表达式执行、动作前置依赖和跨模块同表记录归属。
6. 静态链路保持轻量：业务模块通过平台动作、数据范围能力和 Web 注解接入，不应手写一套权限流程。

## 关联专题

| 专题 | 关系 |
| --- | --- |
| 动静一体核心 | 身份、权限、租户、审计和事务必须被静态/动态模块共享。 |
| 平台配置与元数据 | 模块 alias、菜单挂载和动态动作目录提供授权对象来源。 |
| 动态 Web / OpenAPI | 动态 CRUD、动作执行、动作可用性和 OpenAPI 输出接入权限结果。 |
| 菜单 | 菜单模型和方案由菜单能力维护，身份权限只负责当前用户可见性剪枝。 |
| 审计 | 动作执行和动态审计记录操作者、授权结果和权限动作。 |
| 工作流 | 工作流运行时复用当前用户上下文和数据权限边界。 |
