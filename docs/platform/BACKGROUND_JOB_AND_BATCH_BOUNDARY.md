# 后台任务与异步批次边界

本文用于约束后台任务、批处理、异步导入、补偿任务和长耗时执行的建设边界。当前阶段不建设通用调度中心，也不把批次进度、重试和失败恢复塞进平台运行审计。

同步动作仍优先复用现有平台能力：模块动作、动态记录服务、权限、数据范围、租户、校验、生命周期、事务和运行审计。只有当执行过程需要脱离请求同步完成，或者需要进度查询、重试、暂停恢复、补偿和长期留痕时，才建设专题批次模型。

## 当前已有事实

| 能力 | 当前口径 |
| --- | --- |
| 动态批量动作 | `/{moduleAlias}/{actionCode}/batch` 是同步动作宿主，复用动作权限、数据范围和运行事件链路。 |
| 批量删除 | 标准 `batchDelete` 是同步批量动作，不单独建设批次流水。 |
| 导入导出 | 当前导入执行返回结果、错误文件名和错误文件 token；暂不持久化异步批次。 |
| 记录联动 | 生单返回 `batchId`，来源影响关系可按 batch 查询；这是记录联动专题事实，不是通用后台任务批次。 |
| 编码规则 | 取号重试写入编码 issue log；这是编码专题诊断事实，不是通用调度日志。 |
| 工作流 | 任务、事件、历史和 after-commit 插件归工作流专题，不归通用后台批次。 |
| 配置治理 | 发布、回滚、dry-run 和导入草稿归配置治理事实；多实体 DDL 补偿仍是触发型技术债。 |

## 不建设的能力

当前不建设以下能力：

1. 通用后台任务表、通用任务调度器或统一 worker 集群。
2. 把所有批量动作都包装成异步批次。
3. 在 `RuntimeAuditRecord` 中保存批次进度、逐条错误、重试明细或补偿过程。
4. 对导入、回写、工作流、配置发布等专题强行套同一个批次模型。
5. 无业务边界的通用“任务中心”URL。

## 触发条件

出现以下任一条件时，应按专题建设批次或后台任务模型：

1. 单次执行可能明显超过同步请求可接受时间。
2. 用户需要查询进度、取消、暂停、恢复或下载执行结果。
3. 执行需要自动重试、人工重放、失败补偿或幂等保护。
4. 执行范围跨多个模块、多个实体或多个外部系统，且需要异步编排、补偿恢复或长期解释视图。
5. 任务必须在无登录请求上下文中运行，并需要明确系统态原因和授权边界。
6. 执行明细需要长期保留、审计查询或运维排障。

## 推荐归属

| 场景 | 推荐归属 |
| --- | --- |
| 动态记录同步批量删除、批量动作 | 动态运行态动作链路和运行审计 |
| 业务数据导入异步化 | 导入导出专题批次 |
| 外部系统回调、幂等、重试和回执 | 外部写入接入专题 |
| 生单、回写、来源关系和失败诊断 | 记录联动专题 |
| 工作流异步插件、任务超期扫描和历史事件 | 工作流专题 |
| 配置发布、迁移 dry-run、多实体 DDL 补偿 | 配置治理专题 |
| 平台级计划任务、跨专题补偿和运维重放 | 后续运维治理或后台任务专题 |

## 批次模型草案

触发专题批次建设时，优先使用轻量模型，不预设通用调度中心：

| 字段 | 用途 |
| --- | --- |
| `batchType` | 批次类型，例如 import、external_write、schema_publish。 |
| `sourceModuleAlias` | 业务来源模块。 |
| `triggerActionCode` | 触发动作。 |
| `traceId` | 与运行审计互相定位。 |
| `operatorId` / `systemReason` | 人工触发或系统态触发来源。 |
| `status` | pending、running、success、partial_success、failed、canceled。 |
| `totalCount` / `successCount` / `failedCount` | 执行计数摘要。 |
| `startedAt` / `finishedAt` | 执行时间。 |
| `errorSummary` | 错误摘要，不保存完整业务内容。 |
| `resultRef` | 错误文件、报告或专题结果的引用。 |

逐行数据、外部报文全文、业务字段快照和复杂恢复步骤不作为平台默认字段，应由专题按安全、保留期和查询场景单独设计。

## 与运行审计的关系

运行审计只记录“谁触发了什么动作、授权如何、系统态原因是什么、结果摘要是什么”。批次模型记录“这个长过程后来怎么跑、进度如何、哪些项失败、能否恢复”。两者通过 `traceId`、`batchType`、模块和记录 ID 互相定位，不互相替代。
