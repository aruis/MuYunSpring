# 平台文档

本文档区用于梳理 MuYunSpring 已形成的平台业务能力，帮助内部建设者快速定位能力边界和交接入口。

这里按业务专题组织内容。专题文档只承担线索功能，不复盘推进过程，也不展开 Java 底层设计。

当前目录是平台内部建设和交接的主入口。临时专项记录和已收口清单不长期保留；如果后续发现新的平台缺口，应优先判断它属于哪个业务专题，只有跨专题且尚未形成稳定边界时才临时记录为技术债或专项治理项。

跨专题边界优先放在平台根文档中沉淀。例如 [审计与专题流水边界](AUDIT_AND_PROCESS_LOG_BOUNDARY.md) 用于判断平台运行审计和业务专题流水的职责分工，[外部写入接入边界](EXTERNAL_WRITE_BOUNDARY.md) 用于约束第三方系统、外围平台和后台集成写入平台的方式。

## 阅读方式

每个专题优先包含两类文档：

- `OVERVIEW.md`：说明专题解决什么业务问题、包含哪些主要能力、核心链路和边界。
- `WEB_API.md`：按 URL 维度简要说明 Web 接口用途、关键参数和注意点。

如果专题体量较大，会在专题下继续拆二级目录。二级目录仍遵循相同写法，避免把一篇文档写成过重的设计说明。

## 专题地图

| 专题 | 内容范围 |
| --- | --- |
| `configuration` | 应用、模块、元数据、字段、字典、菜单基础配置，以及配置发布到运行态。 |
| `runtime` | 动态记录运行态、模块描述、CRUD、查询、引用、动作基础和动态 OpenAPI。 |
| `identity-permission` | 租户、组织、用户、角色、登录、动作授权、数据权限、菜单剪枝和当前用户上下文。 |
| `workflow-task` | 工作流定义、发布、实例、审批、任务、分支汇聚、待办、历史、委托和插件挂点。 |
| `workflow-task/workbench` | 待办、已办、知会、发起跟踪、委托视角和工作台统计。 |
| `business-automation` | 平台业务自动化能力的总入口。 |
| `business-automation/code-rule` | 编码规则、预览、取号、台账、回收池和动态记录赋码。 |
| `business-automation/data-exchange` | 导入模板、导入执行、错误文件、导出、子表导出和引用 title 往返。 |
| `business-automation/record-linkage` | 生单、目标草稿、来源关系、回写规则、贡献台账和失败诊断。 |
| `page` | 页面交付与页面交互能力的总入口。 |
| `page/delivery` | 菜单入口、页面 bootstrap、列表查询、汇总、表单保存、附件、查重和引用候选。 |
| `page/interaction` | 关联视图、局部编辑、动态弹窗、动作区块、模块任务和任务检查。 |
| `governance` | 配置包、健康检查、版本发布、回滚、迁移 dry-run、导入草稿和模板复用。 |

## 写作原则

1. 以业务能力梳理为主，少写实现细节。
2. 接口文档按 URL 说明功能点，不追求完整 OpenAPI。
3. 能由代码和测试证明的细节，不在文档中重复展开。
4. 明确当前边界和不承诺项，避免读者误以为阶段性能力已经变成完整产品能力。
