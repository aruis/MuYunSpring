# 外部写入接入边界

本文用于约束第三方系统、外围平台、回调任务或后台集成写入 MuYunSpring 时的接入方式。外部写入不是绕开平台能力的快捷入口，也不是平台运行审计的报文仓库。

当前阶段先明确边界，不建设独立外部接入中心。外部写入如果只是对某个模块执行业务动作，应复用该模块已经开放的标准动作、动态记录写链路、权限、数据范围、校验、生命周期、租户和运行审计。

## 首期口径

1. 外部写入必须落到明确的 `moduleAlias + actionCode` 或模块标准写动作，不开放只按表名、实体名或字段名直写的接口。
2. 外部写入不绕过动态记录服务、静态 service、字段写保护、数据范围、租户上下文和运行审计。
3. 外部系统如果代表某个用户或业务主体执行，应先解决身份映射和授权判断；不能用系统态掩盖业务操作者。
4. 系统态只用于平台可信任务、受控回写和内部补偿。系统态写入必须带明确原因，并进入运行审计。
5. 平台运行审计只记录动作入口、操作者、系统态原因、授权结果、模块、记录和轻量结果，不保存外部报文全文。

## 不建设的能力

当前不建设以下能力：

1. 通用外部写入 URL，例如 `/external/{moduleAlias}/save`。
2. 跳过模块动作授权的数据直写 API。
3. 把外部请求、响应、重试和回执塞进 `RuntimeAuditRecord.payloadText`。
4. 通用报文仓库、通用 webhook 中心或通用事件流。
5. 对所有动态模块自动生成外部系统专用写入契约。

这些能力只有在真实外部接入场景出现后，才按业务边界拆战役建设。

## 推荐接入路径

| 场景 | 推荐路径 | 说明 |
| --- | --- | --- |
| 第三方系统代表用户提交业务单据 | 标准模块动作或动态记录动作 | 先完成身份映射，再按动作权限和数据权限执行。 |
| 内部系统托管回写业务字段 | 记录联动或受控系统态动作 | 复用回写规则、贡献台账和运行审计，不直接更新表。 |
| 定时任务修复或补偿数据 | 后续后台任务调度或运维治理专题 | 系统态原因、批次进度、失败恢复应由专题流水承接。 |
| 外部回调需要幂等和回执 | 后续外部接入专题 | 需要幂等键、请求摘要、状态、回执和失败原因。 |
| 数据批量导入 | 导入导出专题 | 使用导入模板、解析、执行和错误文件链路。 |

## 未来触发条件

出现以下任一场景时，应建设外部接入专题或专门模型：

1. 外部系统需要幂等键、请求摘要、回执状态和重试次数。
2. 外部报文需要按安全策略保存摘要、脱敏内容或下载凭证。
3. 外部写入需要异步执行、失败恢复、补偿或人工重放。
4. 外部身份需要维护应用、密钥、授权范围、限流和吊销。
5. 同一外部来源需要跨模块编排写入，无法自然落到单个模块动作。

## 后续模型草案

触发外部接入专题时，优先考虑轻量模型，不预设重型事件流：

| 字段 | 用途 |
| --- | --- |
| `sourceCode` | 外部来源标识。 |
| `idempotencyKey` | 幂等键，同一来源下唯一。 |
| `moduleAlias` / `actionCode` | 最终落到的平台模块动作。 |
| `recordId` / `traceId` | 与业务记录和运行审计互相定位。 |
| `requestDigest` | 请求摘要或脱敏摘要，不保存完整报文。 |
| `status` | 接收、执行中、成功、失败、已补偿等状态。 |
| `errorCode` / `errorMessage` | 失败定位信息。 |
| `receiptText` | 可返回外部系统的轻量回执。 |

是否保存报文全文、保存多久、谁能下载，应由安全、合规和具体业务共同决定，不作为平台默认能力。
