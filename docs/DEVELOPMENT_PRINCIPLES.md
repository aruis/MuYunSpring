# 开发原则

本文记录当前阶段已经确认的开发原则。它约束方向，不替代测试和代码实现。

## 核心路线

MuYunSpring 的目标是形成“动静一体”的平台：静态 Java 模型和动态元数据模型共享同一套能力、数据访问、生命周期和治理语义。

业务模块的理想形态是声明自己拥有哪些平台能力，而不是重复编写平台流程：

```java
@Service
public class OrganizationService implements
        CrudAbility<Organization>,
        SoftDeleteAbility<Organization>,
        TreeAbility<Organization>,
        ReferenceAbility<Organization> {
}
```

`TreeAbility` 已经包含同级排序语义。树形业务不需要再额外声明 `SortAbility`，避免把稳定业务层级误拆成可选 mixin。

## 取舍原则

1. 保留能力组合、标准模型、标准 DAO、生命周期 hook、静态/动态统一运行时这些核心形态。
2. 避免含义不清的命名、冗余字段、隐式扩展 map 和过长的 service hook。
3. 普通业务接入应足够轻：模型、DAO、Ability、少量 hook、测试证明即可。
4. 平台基础设施可以多走一步，尤其是数据操作、建表、元数据、生命周期和能力组合边界。
5. 不为了拆分而拆分。只有当拆分能减少依赖混乱、复用成本或认知负担时才新增包或模块。

## 推进方式

1. 按能力单元或业务单元推进，不追求过小步骤。
2. 一个能力单元应尽量包含公开契约、默认实现、测试证据和必要文档更新。
3. 文档写方向、边界、验收口径；细节行为优先由测试表达。
4. 发现实现别扭时，先判断是不是平台底座抽象不足，不急着在业务层硬编码补丁。
5. 每个阶段都要复盘是否仍在逼近“动静一体”和“代码量压缩”两个目标。

## 当前非目标

M1 阶段不展开菜单、权限、登录、工作流、编码规则、导入导出、生成、回写等应用级能力实现。只有当这些能力暴露出必须提前下沉的底层契约时，才允许进入当前阶段。
