# MuYunSpring 里程碑规划

本文用于约束新 MuYun 的阶段推进节奏。当前核心路线是“动静一体”：静态 Java 模型和动态元数据模型不应演化成两套平台，而要共享同一套模型基座、数据访问、能力组合和生命周期语义。

文档详细展开已完成的 M1、M1.5、M2、M3、M4、M5、M6、M7 和当前收口的 M8。后续里程碑只保留阶段目标和进入条件，等前置里程碑完成后再补功能点矩阵，避免过早设计造成文档负担。

## M1 动静一体核心底座

M1 的目标不是先做一批静态业务，也不是先做完整动态平台，而是先证明两类模型能共享一套平台底座。

### 验收目标

1. 静态模型通过 Java 类、注解、DAO 和 Ability 组合获得标准平台能力。
2. 动态模型通过元数据定义实体、字段和表结构，并复用同一套能力语义。
3. 静态 DAO 和动态 DAO 在数据操作层共享一致的查询、分页、排序、软删、版本和生命周期规则。
4. 静态模型自动建表与运行态元数据建表都具备最小闭环。
5. 至少一个静态样例和一个动态样例能证明 CRUD、树、排序、引用等基础能力语义一致。

### M1 收尾结论

1. M1 的必要验收已完成：静态模型和动态元数据模型已经能接入同一套基础能力、数据访问、生命周期、租户、软删、版本、树、排序、引用、启停、缓存和父子聚合语义。
2. 真实数据库行为测试已覆盖静态仓储入口、动态 CRUD、动态稳定门面、树、排序、引用标题、引用投影、父子聚合、租户隔离、租户内唯一和复杂 Criteria 组合。
3. M1 不再保留阻塞项。跨节点缓存治理、身份体系租户来源、动态 schema 多实体失败补偿、显式 HYDRATED 深度读取等进入 M1.5 或后续阶段。
4. 后续每补一个能力或门面，仍必须同步检查静态/动态两侧，避免重新分裂成两套平台。

### 推进原则

1. 按能力单元推进，而不是按零散小点推进。一个能力单元应尽量包含模型、接口、默认实现、测试和必要的真实数据库验证。
2. 代码、测试、注释、文档各司其职。能由测试表达的细节，不写成长篇设计文档。
3. 平台基础设施允许多走一步。涉及数据操作、能力组合、生命周期、建表和元数据边界时，应优先考虑中长期复用。
4. 不为了拆分而拆分。包和模块边界只有在降低心智负担、减少重复、稳定依赖方向时才继续拆。
5. M1 和 M1.5 期间不展开菜单、权限、登录、工作流等平台应用能力，除非它们暴露出必须提前下沉的底层契约。

## M1.5 能力层夯实

M1.5 的目标是在进入平台标准业务前，把基础能力从“主链路可用”推进到“边界清晰、默认好用、能支撑中期业务接入”。这一阶段可以吸收成熟平台的优秀做法，但产物必须沉淀为 MuYun 自己的命名、模型和契约。

### 验收目标

1. 生命周期拆出平台内部链和业务扩展链，平台能力不依赖业务 hook 手动调用 `super` 才能正确执行。已在 CRUD 与父子聚合链路完成第一轮收口。
2. 引用能力从标题读取升级为统一引用定义、投影、默认 RAW 加载边界和跨引用缓存失效基础。
3. 缓存能力具备 namespace 生命周期、容量或过期边界、动态运行态发布/卸载清理入口。已完成本地默认边界，后续只在需要监控、跨节点一致性或业务级策略时升级为缓存管理器。
4. 静态和动态写入形成统一乐观锁 CAS contract，旧版本更新和删除必须有明确失败语义。能力层、动态 DAO 和静态 Repository 已统一走底层条件写入口；静态删除走条件删除，动态软删走条件更新。
5. 动态 schema 演进覆盖字段新增、约束变更、字段删除、dry-run 和单实体失败不污染运行态；多实体部分执行补偿进入发布中心设计。
6. Criteria 真实数据库 contract 已覆盖复杂组合、分页排序、非法字段拒绝和 raw 条件边界；后续按新查询能力继续增量扩展。
7. 技术债已按 M2 配置自举、M3 动态运行时、场景触发和运维治理分层；M1.5 基础能力没有必须阻塞 M2 的技术债。
8. 静态 service 和动态门面已经对齐基础查询入口，业务侧可直接通过能力层 `list`、`pageQuery`、`count` 获取统一租户和软删 scope。
9. 每个能力补强都同时检查静态模型和动态模型，避免重新分裂成两套平台。

M1.5 完成前，不以扩展业务线数量作为主要进度指标。

## M2 平台配置与元数据自举

M2 的目标是让平台先能管理自身配置态。第一批聚焦应用、模块、元数据、菜单和数据字典；组织、部门、用户、角色顺延到后续里程碑。M2 不追求完整管理后台 UI，而是证明配置业务能复用能力体系，并能从持久化配置发布到动态运行态。

### 验收目标

1. 应用、模块、元数据、菜单、数据字典使用统一 alias 规则，模块身份统一使用 `moduleAlias`，不引入 `moduleId` 同义词。
2. 模块 alias 必须满足 `applicationAlias.xxx`，创建后不允许手动修改；静态模块可以没有元数据，动态模块必须有一个主元数据。
3. 元数据 `id` 是平台稳定 ID，`metadataAlias` 在应用下唯一，物理表定位使用 `schemaName + tableName`。
4. 模块与元数据通过关系绑定，一个模块最多一个主元数据，关系模型能表达父、子、孙结构。
5. 跨模块体现时，菜单、入口、动作、查询、权限、运行时 URL 和审计口径统一使用当前模块的 `moduleAlias`。
6. 至少跑通一条 `应用 -> 模块 -> 元数据 -> 发布 -> DynamicRecordService.entity(moduleAlias, metadataAlias) -> CRUD/query` 闭环。

## M3 动态模块可交付闭环

M3 的目标是让动态模块从基础数据能力推进到运行时交付能力。前半段已经完成字段字典、动态查询、动态引用展示、运行态 descriptor 和基础视图输出，但这只说明动态模块具备“结构可描述”的基础，不等于已经具备完整交付能力。

校准后的 M3 继续聚焦动态平台基础能力，不提前进入业务线。M3 完成时，一个动态模块应至少能被前端或调用方按稳定契约使用：能读取模块结构，能得到列表/表单配置，能解析引用，能看见可执行动作，并且这些配置有后续权限、审计、工作流、导入导出等能力的挂载位置。

M3 不追求完整 UI 设计器，也不在本阶段完成权限、工作流、导入导出、编码规则等应用级能力；这些能力按依赖关系进入后续里程碑。但 M3 必须先把视图、引用、动作和字段行为的基础契约整理清楚，避免后续继续堆临时接口。

## M4 平台执行底座

M4 的目标不是直接铺开所有应用专题，而是在 M3 动态模块可交付边界稳定后，先补齐后续平台应用能力共同依赖的执行底座。规则、动作、事件、审计、时间字段和 Web/OpenAPI 交付如果没有统一口径，编码规则、导入导出、工作流、推送生成和回写都会被迫各自发明临时机制。

M4 的规则与公式不做简化占位版，而应按可长期承载字段计算、保存校验、条件判断、动作可用性、导入校验和工作流条件的正式引擎推进。确有复杂能力可以分批落地，但模型、执行上下文、错误报告和扩展边界必须按完整闭环设计。

## M5 权限与身份闭环

M5 的目标是把身份、角色、动作权限、数据权限和菜单剪枝做成完整闭环。权限不做“只留一个权限字符串字段”的半成品；如果进入实现，就必须能支撑动作执行、动态模块、静态模块、菜单、数据范围和审计上下文。

M5 功能点矩阵在 M4 完成后展开。当前先固定以下边界：

1. 身份上下文至少包括用户、租户、组织、部门、角色和系统态操作者。
2. 动作权限要能同时约束静态模块动作和动态模块动作。
3. 数据权限是独立授权策略，不和 `tenantId` 基线隔离混淆。
4. 菜单权限是菜单树剪枝，不改变菜单本身的租户隔离和方案模型。
5. 权限判定结果要能进入动作执行上下文和审计上下文。

### M5 收口结论

M5 已达到阶段收口条件：身份上下文、租户上下文、角色授权、动作执行、数据权限、菜单剪枝、静态模块自注册、动态模块动作接入和轻量审计上下文已经形成同一条后端闭环。

M5 不再继续扩展管理端 UI、复杂自定义数据权限 DSL、动作前置依赖、跨模块同表记录归属和专题业务流水。后续如果真实业务需要这些能力，按对应专题进入 M6/M7 或技术债回收，不再阻塞 M6 启动。

本阶段明确后移或暂不扩展的事项：

1. 自定义数据权限条件暂不开放 SQL/DSL 执行，只保留模型字段和授权入口 fail-fast 边界，等真实场景出现后再做安全表达式设计。
2. 跨模块共用同一物理表时的记录权限归属字段暂不进入 M5 P0。当前模块入口统一按当前 `moduleAlias` 判权；若后续出现同表多业务归属并存，再补“记录归属模块”能力。
3. 动作前置依赖暂不进入 M5。当前 `authInheritActionCode` 只表达未命中时的 fallback，不表达“必须同时具备其它动作权限”的 AND 语义；若后续需要“能删必须先能看/能改”，应单独设计模型和记录级数据范围校验。
4. 静态业务专题流水不并入平台动作审计。平台审计只记录必要动作上下文，工作流、导入导出等专题自行维护业务流水。
5. 管理端 UI 编排不进入 M5。M5 只保证模型、服务、Web contract 和测试证据可支撑后续 UI。

## M6 工作流与任务闭环

M6 的目标是将审批节点和任务节点统一为工作流能力，并形成完整闭环。工作流不做只有状态字段和待办入口的简化版；进入实现后，应覆盖定义、发布、运行、任务、待办、历史、撤回、终止、回退、转办、加签、分支、汇聚和插件挂点等核心能力。

M6 采用统一 HYBRID 工作流模型，不拆成审批流和任务流两套实现。流程通过 `approvalEnabled` 声明是否拥有业务记录主审批态治理权；启用审批治理的流程在同一模块、租户、业务记录上当前有效实例独占，并通过 `APPROVAL_COMPLETED` 里程碑回写审批通过。审批完成不等于整条流程结束，后续仍可继续执行任务、系统动作或通知等节点。

M6 吸收成熟审批与任务业务规则，并使用 MuYun 自己的 workflow 命名、动静一体模型、M4 动作/事件/审计上下文和 M5 身份/权限闭环。

### M6 边界

1. 工作流节点包括审批节点和任务节点，任务是可独立配置、判定和引导动作的业务对象。
2. 工作流动作必须接入 M4 的动作执行上下文、规则引擎、事件和审计上下文。
3. 工作流权限必须接入 M5 的身份与权限闭环。
4. 工作流状态字段、实例字段和任务字段要兼顾静态模型与动态元数据模型。
5. `approvalEnabled=true` 的流程负责业务记录审批态；`approvalEnabled=false` 的流程仍可运行任务闭环，但不回写主审批态。
6. `APPROVAL_COMPLETED` 之后的节点不再反向影响主审批态。普通撤回、驳回、回退默认不跨越审批完成边界；管理端重置或强制终止另行治理。
7. 超期能力只做预警和超期标记，不自动审批或自动流转。
8. 任务节点完成必须以后端判定或人工确认收口为准，前端交互完成不等于任务完成。
9. 分支和汇聚必须具备独立 route 运行对象，不能只靠节点状态推导有效路径。
10. 任务定义、任务引导和任务完成判定应作为平台能力沉淀，支持静态与动态业务共同接入。

### M6 收口边界

M6 以稳定后端闭环收口，不继续为了兼容旧系统复制字段、接口或历史包袱。已确认不阻塞 M6 的后续扩展包括：

1. 业务记录标题：MuYun 当前有引用标题能力，但没有统一的跨模块 `moduleAlias + recordId -> recordTitle` 门面；工作台和历史不为兼容 neo 的 `title` 字段硬补，后续如需要应建设 MuYun 自身的业务记录标题 resolver。
2. 复杂参与人策略：当前运行边界支持用户参与人解析并保持任务生成确定性；角色、部门、相对人、表达式参与人等策略后续按真实组织模型和权限上下文扩展。
3. 复杂回退：当前支持线性回退和有效分支域回退到分支前节点；嵌套分支、汇聚后回退等复杂路径继续 fail-fast，按具体流程场景补测试后扩展。
4. 前端 UI 编排：流程绘制所需的语义图和布局图 JSON 已由后端保存、冻结和随 render bundle 返回；MANUAL 分支候选、工作台展示增强、委托管理和菜单编排已有后端读写入口，完整 UI 不作为 M6 后端收口条件。
5. 插件扩展：M6 固定同步和 after-commit 插件调度边界；异步分发、插件配置管理、动态脚本和 addSign 图编辑插件后续另设专题。
6. 模块动作回收：workflow 发布后贡献的 runtime 动作目录保持模块级稳定可授权，不随单个定义停用自动回收；如业务需要精细回收，再进入动作目录治理专题。

## M7 平台业务编排能力

M7 的目标是推进编码规则、导入导出、推送生成和回写等可复用平台应用能力。其中推送生成和回写业务复杂度高，明确后置到本阶段，不提前塞进 M4/M5/M6。

M7 已在 M6 后端闭环之后完成阶段收口。当前推进方式不是重新从零设计平台应用能力，而是按业务单元吸收成熟系统经验，再沉淀为 MuYun 自己的动静一体语义、Web 暴露面和测试契约。编码规则、导入导出、推送生成和回写已按动态表单闭环推进到稳定边界，后续增强按明确业务场景单独进入专题。

当前固定以下顺序：

1. 编码规则优先，依赖规则引擎、动作上下文、事件和事务边界。
2. 导入导出其次，依赖 descriptor、字段行为、字典、引用、时间字段和规则校验。
3. 推送生成与回写最后推进，依赖规则、动作权限、系统态写入、事件、审计和必要的工作流挂点。

### M7 当前推进口径

编码规则当前按 M7 第一批能力收口。实现已覆盖平台规则模型、预览、正式生成、序列状态、台账、回收池、动态记录 create/update/delete 生命周期、主子记录上下文、联动依赖、业务时间、组织规则解析、旧码释放 fallback 分级和运行职责拆分。序列分配已通过 PostgreSQL 原子 upsert 路径验证同桶并发不重复和外层事务回滚；回收池消费已通过 PostgreSQL 行级锁与 `skip locked` 路径验证同桶并发只消费一次。非 PostgreSQL 数据库还不能宣称具备同等原子语义。动态自动赋码已通过真实动态发布和记录创建路径验证，规则配置与业务记录按当前租户上下文生效，系统态规则模板是否跨租户下发后续单独设计。

编码台账是当前占用事实表，不是流水日志；同一租户、同一规则、同一编码值只有一条当前事实，状态在 `ACTIVE`、`AVAILABLE`、`DISCARDED` 间流转。可回收编码字段不能依赖普通动态字段唯一索引表达唯一性，否则软删后的编码复用会被数据库字段索引挡住；当前唯一占用由编码台账和动态生命周期校验承担。规则生成服务本身保持静态/动态中立，静态业务必须先注册进平台模块体系，再通过 `CodeRuntimeFacade` 显式调用目标 `moduleAlias/entityAlias/fieldName` 的动态配置规则；`generateOnly` 只生成不占用，`issueAndBind` 在事务边界内生成并绑定 sourceRecordId，重复绑定同一 sourceRecordId 幂等返回当前 active 绑定。静态业务 A 调用动态业务 B 的编码规则属于“调用 B 的规则而非 B 的记录服务”，只要 A 提供满足 B 规则依赖的 context 即可。静态 service 的自动赋码声明与生命周期挂载尚未作为最终契约开放，后续应在静态模型字段声明、业务 hook 覆盖边界和真实事务测试明确后再补，不把当前动态自动接入包装成最终形态。

导入导出当前按动态模块交换能力阶段收口，不继续深挖非必要增强。已完成模板请求配置、引用下拉、引用 title 反推、引用 title 导出、历史错误文件、子表导出、Web/OpenAPI 暴露面和 EXCHANGE 能力门控。业务引用作为模板的一部分时，模板可默认生成辅助下拉，也允许按字段关闭；不论是否生成下拉，导入都应支持 title 反推。重复 title 不能生成看似可选但无法稳定反推的下拉项。后续若需要引用 projection、跨 sheet 辅助页、静态业务导入导出或更复杂子表层级，应按明确业务场景单独进入增强项，不阻塞 M7 编码规则主线。

推送生成和回写按动态表单闭环吸收成熟业务颗粒度，但不复制旧系统的包结构、历史字段或临时前端协议。推送生成首期的稳定边界是“源记录生成目标记录草稿”：规则配置归属于源动态模块，启用后贡献记录级 `GENERATE` 托管动作；动作目录只承载托管动作和权限入口，生成条件不作为动作可用性表达式下发；执行时先按动态 action 链路校验源记录动作权限，生成条件按持久化源记录快照判断，未命中时返回空草稿且不额外校验目标模块新增权限，命中时再校验目标模块新增权限；运行结果返回目标模块草稿、`batchId` 和每个草稿的 `originContext`，不直接创建目标记录、不登记执行历史、不限制生成次数、不触发审批自动流转。目标模块后续通过原生新增或生成草稿确认入口提交草稿，保存成功后若请求根携带 `originContext`，平台在同一事务内登记最终来源影响关系。引用字段生单通过 `ReferenceRecordGenerationFacade` 和动态 Web 入口按当前模块、实体、字段、选中引用记录触发，调用方不直接拼规则 action；引用候选解析支持跨动态模块目标，生成草稿确认入口接受草稿 record 的一级子表 `children` envelope。`originContext` 是业务来源声明，不是安全 token，也不能写入动态业务字段、扩展 map 或子表行；运行态只把它作为动态保存的 mutation metadata 传递。

来源影响关系是推送生成之后继续支撑任务判定、生成关系查询和后续回写定位的公共事实表，不属于生成规则的临时会话。首期只承诺动态根记录之间的主动生成影响关系；查询门面已支持按 source、target、batch、generationRule、source+target+rule 查询，并提供已生成判断，后续模块完成项中的 `GENERATED_QUERY_EXISTS` 应基于该事实判断。运行态已从单根对象映射推进到“根草稿 + 已加载一级子表草稿”映射，子 mapping 通过 source/target relation code 从源记录 children 取行并写入目标草稿 children，不直接保存目标记录，也不为子行单独登记影响关系。分录拆分首期支持按一个已加载子表 splitDriver 逐行生成多个目标根草稿，可把明细字段映射到目标根，也可为每个目标根草稿附带一个目标子行；当前明确拒绝根对象 splitDriver、多个 splitDriver、数量拆分、分组聚合和多子 mapping 组合。推送生成与回写的联合动态 contract 已覆盖“源记录执行生成动作 -> 返回目标草稿 -> 目标保存登记来源关系 -> 生成关系定位回写源记录”的主闭环。多根草稿、源子表自动加载、子行级来源事实和跨层级子表映射后续按业务场景开放。引用拉单、链路图谱、前端审计页、草稿 token、防伪签名和跨系统最终一致补偿后续按明确场景扩展。若后续建设模块完成项，应只表达模块级完成定义、引导和后端可验证 check，不能把模块完成项做成平行于 M6 workflow task 的待办系统。

回写按独立战役推进，不能用推送生成的托管动作或目标草稿机制替代。回写的核心闭环是“业务事件 -> 条件状态机 -> 目标定位 -> 字段影响 -> 受控系统态保存 -> ExecutionLog/EffectLog”。当前已先补齐动态写事件快照和同步阻断入口：新增保存事件提供 after，更新保存事件提供 before/after，删除事件提供 before；回写保存可以通过 `WRITE_BACK` 来源、trace、depth、parentExecutionId 和 cascade 策略进入动态保存链路，并继续触发目标模块保存事件。规则执行首期开放动态主记录到主记录的等值定位、生成关系定位、状态触发与字段写入；一层目标子表行定位已支持在规则上声明 `targetRelationCode/targetEntityAlias`，match rule 为空关系时定位目标根，match rule 指向目标关系时定位目标子行，最终通过目标根记录的局部子表受控保存链路更新命中子行，并保留 relation child lifecycle。状态触发支持 `ALWAYS`、进入生效状态、退出生效状态和生效期间来源值变化，并允许同一规则组合多个状态触发模式，当前以单个动态字段等值作为生效判断。字段写入支持 `COVER` 覆盖以及 `ADD/SUBTRACT` 数值贡献，`ADD/SUBTRACT` 必须绑定状态触发，进入时写全量贡献，退出时基于 active EffectLog 撤销历史贡献，生效期间变化时基于 active EffectLog 只写新旧贡献差额；非贡献型字段已基于 APPLIED EffectLog 识别同 rule、触发记录、目标记录、目标字段、来源字段和 afterValue 的重复执行并转为 no-op；目标保存成功后才写入新 effect 并反转旧 active effect，目标保存失败不提前污染贡献事实。重复进入生效、重复退出生效等重复事件首期通过 active 贡献状态转为 no-op，不重复修改目标字段。生成关系定位基于来源影响关系从已生成目标记录回到源记录，适用于“源单推送生成目标单，目标单保存后回写源单”的动态闭环。当前目标 0 命中或多命中仍 fail-fast 并阻断来源事务，多目标策略和 no-op 语义后续按业务场景补齐；该闭环已用真实动态保存入口锁定“源动态记录保存 -> 规则执行 -> 目标动态记录受控保存”、“源动态记录状态变化 -> 数值贡献回写/撤销”、“生成目标动态记录保存 -> 来源影响关系定位 -> 源动态记录受控保存”和“源动态记录保存 -> 目标子表行定位 -> 目标根局部子表受控保存”的契约。ExecutionLog 当前提供按 trace、rule、触发记录、目标记录和 status 的诊断查询；planned、success 和 no-op 状态随业务事务提交，目标保存失败时通过独立事务保留已定位目标和字段 patch 快照。EffectLog 当前记录目标保存成功后的字段级影响事实，包括 execution、trace、触发记录、目标记录、目标字段、来源字段、操作、状态、贡献值、数值 delta 和 before/after 文本值；`ADD/SUBTRACT` 首期已具备 ACTIVE/REVERSED 贡献状态和状态驱动的重复写入保护，但尚未承诺数据库级唯一幂等。默认 `SINGLE_HOP` 不消费由回写引发的下游回写事件，只有显式 `CASCADE` 才允许链式回写。源子表触发、多目标策略、审批事件和数据库级强幂等唯一约束后续继续补齐。

## M8 低代码页面交付闭环

M8 的目标不是提前铺具体业务专题，也不是先做完整设计器，而是把动态模块从“后端可交付”推进到“低代码页面可交付”。本阶段聚焦动态表单页面闭环：字段 UI 类型、UI 配置、查询模板、菜单入口、页面 bootstrap、列表查询、详情表单、子表、引用、汇总和保存辅助能力要形成一条稳定运行链，让前端不需要硬编码模块结构、查询表单、列表列、表单字段、入口上下文和常见页面辅助逻辑。

M8 参考 neo 已实现的成熟业务颗粒度：字段 UI 类型目录、UI Set/UI Config、查询模板与查询项树、模块入口、动态保存里的 `uiConfigId`、引用候选列表里的目标 `uiConfigId/queryTemplateId`、引用解析、汇总面板、查重、附件、乐观锁、列表导航、用户偏好和模块 OpenAPI 文档入口都需要合理吸收；但 MuYun 不复制 neo 的历史表名、包结构、字段袋和临时协议，应按当前 `moduleAlias`、`ModuleMetadataField.id`、运行态 descriptor、动作目录、权限菜单剪枝、发布快照和动态 Web 契约重建。

附件接入以业务对象和 `fileId` 的关系为 MuYunSpring 责任，文件二进制、文件元数据、查看和下载能力归属 `MuYunFileServer`；业务关系表不复制 fileserver 的 MIME、大小、上传人、上传时间等文件事实。

### M8 固定边界

1. M8 聚焦动态低代码页面交付，不回头扩静态表单业务。
2. UI 配置以模块为入口，以元数据字段稳定 ID 为字段锚点，不再引入与 `ModuleMetadataField.id` 平行的字段身份。
3. 查询配置要复用 M3 动态查询、M5 数据权限和 M7 引用/生成/回写上下文，不单独发明查询执行链。
4. 菜单入口只负责定位模块、页面模式、默认 UI 配置、默认查询方案和默认动作上下文；权限仍由 M5 菜单剪枝和动作授权负责。
5. 菜单驱动页面入口对外使用 `/platform.menu/{menuId}/entry`，按 `clientType` 解析 WEB/APP 配置，公共响应以 resolved 页面契约为主，不把内部发布快照作为前端长期依赖；当前不复制 neo 的独立 `ModuleEnter` 模型，等非菜单入口真实出现后再抽象为平台入口模型。
6. 汇总面板、引用候选、查重、附件、列表导航、用户偏好和 OpenAPI 文档入口属于页面可交付基础能力，进入 M8；关联视图、局部信息编辑、动态弹窗和模块任务进入 M9。
7. M8 先稳定后端配置模型、发布契约、运行态输出和 contract/IT 测试；完整可视化设计器、复杂拖拽布局和业务专题页面进入后续阶段。

### M8 当前推进口径

M8 应按“页面可交付闭环”拆战役，而不是按零散模型文件拆。

1. 配置底座战役：字段 UI 类型目录、UI Set、模块 UI 配置模型、查询模板、查询项树、配置校验和发布快照一起收口，确保字段身份、字段类型、UI 控件、配置明细和 Criteria 编译稳定。
2. 菜单入口与页面 bootstrap 战役：把菜单入口、模块入口、默认 UI 配置、默认查询模板和页面初始化包连起来，前端按菜单或模块即可获得页面运行上下文。
3. 动态页面运行态战役：把列表、查询、详情、表单、子表、引用候选、引用解析、summary panel、动作按钮和导出可复用查询上下文串成端到端运行链。
4. 页面保存与交付辅助战役：补齐 UI 必填保存、查重、乐观锁、附件、列表导航、用户偏好和模块 OpenAPI 文档入口，让动态页面具备真实交付体验。

动态弹窗、复杂关联视图、局部信息编辑、模块任务和设计器体验只在 M8 预留挂点，不在页面主链路稳定前扩散。

### M8 收口结论

M8 第一批页面交付能力已收口。动态低代码页面已具备配置模型、发布快照、菜单入口、bootstrap、查询与汇总、保存校验、附件关系、查重预检、引用候选、列表导航、用户偏好、模块 OpenAPI 和稳定 contract 基线。M9 可以在此基础上推进关联视图、局部信息编辑、动态弹窗、模块任务和业务交互编排，不再回到 M8 的基础页面契约上扩散范围。

M8 与 neo 颗粒度对齐后形成的补充治理项已回收到 [M8 稳定契约](M8_STABLE_CONTRACT.md) 和本里程碑状态说明；[M8 临时专项治理](M8_TEMPORARY_GOVERNANCE.md) 仅保留为阶段性回看索引，不再作为并行待办路线。

## M9 低代码交互编排与业务动作 UI 化

M9 在 M8 页面交付闭环稳定后启动，目标是把页面之外的业务交互和动作执行做成可配置能力，而不是让前端按模块写专用页面。M9 应吸收 neo 关联视图、模块上下游关系、模块信息编辑、动态弹窗、模块任务完成项、动作上下文和 M7 业务编排诊断的成熟颗粒，但仍以 MuYun 的动作目录、权限、动态 Web、生成/回写事实、工作流任务和运行审计为底座。

M9 当前按三个战役推进：

1. 关联视图与局部信息编辑战役：建设模块上下游关系、关联视图、关联查询映射、详情关联区块和局部信息编辑动作。关联视图复用 M8 查询模板和 M7 生成关系事实，信息编辑必须作为独立动作走动作授权、字段范围校验、UI 配置保存和乐观锁，不退化成标准 edit 的展示变体。当前已启动关联视图查询入口，详情页可按 association view code 查询子表关联列表或引用目标记录页，并复用 M8 查询请求协议；详情 UI 配置可通过 `layoutJson.blocks[].type=associationView` 声明关联区块，bootstrap 下发 `associationBlocks`、目标 UI 配置、默认查询模板和查询路径；局部信息编辑通过内置 `muyun.localEdit` 动态 action executor 承载，动作自身负责授权，执行器按发布 UI 配置限制主表字段范围并复用动态乐观锁更新。
2. 动态弹窗与动作交互战役：建设动态弹窗动作的初始化与提交协议、动作按钮位置与可用性、动作前后刷新/跳转策略和前端状态刷新契约。动态弹窗只承载“先补参数再执行业务”的动作交互，不作为任意脚本、任意组件注册或完整页面引擎。当前已启动弹窗交互协议：DIALOG action 作为弹窗初始化入口，`executorKey` 可用 `dialogKey#submitActionCode` 绑定后续业务动作；初始化结果返回 `dialogKey/actionCode/submitActionCode/submitPath/recordId/refreshOnSuccess`，提交仍走动态 action 主链路；默认 UI 配置可通过 `layoutJson.blocks[].type=dialog|action|localEdit` 声明动作区块，bootstrap 下发 `actionBlocks` 供前端结合已授权动作和可用性渲染按钮。
3. 模块任务与业务完成项战役：建设模块级业务完成项、默认引导、后端可验证 check、托管来源同步和业务编排诊断。模块任务不是审批待办，不存处理人、委托、加签或办理状态；完成判定应复用公式、查询模板、关联视图、生成关系、审批状态和工作流上下文，不新增平行条件语言。

M9 不做完整 BPMN 设计器，也不把弹窗 handler、信息编辑或模块任务变成业务硬编码入口；所有动作必须能回到平台动作上下文、权限判定和配置化运行态。

## M10 低代码生产化治理与业务专题承接

M10 在 M8-M9 证明低代码页面和交互编排可支撑真实业务交付后启动，目标是从“能配能跑”推进到“可治理、可迁移、可复用、可承接业务专题”。旧版 M8 的“业务专题覆盖”后移到 M10：只有当模块页面、查询、动作交互和发布治理稳定后，才按具体业务单元推进，避免业务代码绕开平台能力。

M10 候选能力包括 UI/查询/动作配置版本管理、发布审批、回滚、租户模板、配置导入导出、跨环境迁移、配置差异比对、低代码样板模块、配置器体验治理、性能与缓存治理、运行态观测、配置健康检查、存量配置迁移审计，以及第一批真实业务专题的低代码落地。组织、部门、用户、角色等身份基础能力仍归 M5，不作为 M10 普通业务专题重复建设。
