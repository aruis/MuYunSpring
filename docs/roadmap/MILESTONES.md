# MuYunSpring 里程碑规划

本文用于约束新 MuYun 的阶段推进节奏。当前核心路线是“动静一体”：静态 Java 模型和动态元数据模型不应演化成两套平台，而要共享同一套模型基座、数据访问、能力组合和生命周期语义。

文档详细展开已完成的 M1、M1.5、M2 以及当前推进的 M3。后续里程碑只保留阶段目标和进入条件，等前置里程碑完成后再补功能点矩阵，避免过早设计造成文档负担。

## M1 动静一体核心底座

M1 的目标不是先做一批静态业务，也不是先做完整动态平台，而是先证明两类模型能共享一套平台底座。

### 验收目标

1. 静态模型通过 Java 类、注解、DAO 和 Ability 组合获得标准平台能力。
2. 动态模型通过元数据定义实体、字段和表结构，并复用同一套能力语义。
3. 静态 DAO 和动态 DAO 在数据操作层共享一致的查询、分页、排序、软删、版本和生命周期规则。
4. 静态模型自动建表与运行态元数据建表都具备最小闭环。
5. 至少一个静态样例和一个动态样例能证明 CRUD、树、排序、引用等基础能力语义一致。

### M1 收尾结论

1. M1 的必要验收已完成：静态模型和动态元数据模型已经能接入同一套基础能力、数据访问、生命周期、租户、软删、版本、树、排序、引用、启停、缓存和父子聚合语义。
2. 真实数据库行为测试已覆盖静态仓储入口、动态 CRUD、动态稳定门面、树、排序、引用标题、引用投影、父子聚合、租户隔离、租户内唯一和复杂 Criteria 组合。
3. M1 不再保留阻塞项。跨节点缓存治理、身份体系租户来源、动态 schema 多实体失败补偿、显式 HYDRATED 深度读取等进入 M1.5 或后续阶段。
4. 后续每补一个能力或门面，仍必须同步检查静态/动态两侧，避免重新分裂成两套平台。

### 推进原则

1. 按能力单元推进，而不是按零散小点推进。一个能力单元应尽量包含模型、接口、默认实现、测试和必要的真实数据库验证。
2. 代码、测试、注释、文档各司其职。能由测试表达的细节，不写成长篇设计文档。
3. 平台基础设施允许多走一步。涉及数据操作、能力组合、生命周期、建表和元数据边界时，应优先考虑中长期复用。
4. 不为了拆分而拆分。包和模块边界只有在降低心智负担、减少重复、稳定依赖方向时才继续拆。
5. M1 和 M1.5 期间不展开菜单、权限、登录、工作流等平台应用能力，除非它们暴露出必须提前下沉的底层契约。

## M1.5 能力层夯实

M1.5 的目标是在进入平台标准业务前，把基础能力从“主链路可用”推进到“边界清晰、默认好用、能支撑中期业务接入”。这一阶段可以吸收成熟平台的优秀做法，但产物必须沉淀为 MuYun 自己的命名、模型和契约。

### 验收目标

1. 生命周期拆出平台内部链和业务扩展链，平台能力不依赖业务 hook 手动调用 `super` 才能正确执行。已在 CRUD 与父子聚合链路完成第一轮收口。
2. 引用能力从标题读取升级为统一引用定义、投影、默认 RAW 加载边界和跨引用缓存失效基础。
3. 缓存能力具备 namespace 生命周期、容量或过期边界、动态运行态发布/卸载清理入口。已完成本地默认边界，后续只在需要监控、跨节点一致性或业务级策略时升级为缓存管理器。
4. 静态和动态写入形成统一乐观锁 CAS contract，旧版本更新和删除必须有明确失败语义。能力层、动态 DAO 和静态 Repository 已统一走底层条件写入口；静态删除走条件删除，动态软删走条件更新。
5. 动态 schema 演进覆盖字段新增、约束变更、字段删除、dry-run 和单实体失败不污染运行态；多实体部分执行补偿进入发布中心设计。
6. Criteria 真实数据库 contract 已覆盖复杂组合、分页排序、非法字段拒绝和 raw 条件边界；后续按新查询能力继续增量扩展。
7. 技术债已按 M2 配置自举、M3 动态运行时、场景触发和运维治理分层；M1.5 基础能力没有必须阻塞 M2 的技术债。
8. 静态 service 和动态门面已经对齐基础查询入口，业务侧可直接通过能力层 `list`、`pageQuery`、`count` 获取统一租户和软删 scope。
9. 每个能力补强都同时检查静态模型和动态模型，避免重新分裂成两套平台。

M1.5 完成前，不以扩展业务线数量作为主要进度指标。

## M2 平台配置与元数据自举

M2 的目标是让平台先能管理自身配置态。第一批聚焦应用、模块、元数据、菜单和数据字典；组织、部门、用户、角色顺延到后续里程碑。M2 不追求完整管理后台 UI，而是证明配置业务能复用能力体系，并能从持久化配置发布到动态运行态。

### 验收目标

1. 应用、模块、元数据、菜单、数据字典使用统一 alias 规则，模块身份统一使用 `moduleAlias`，不引入 `moduleId` 同义词。
2. 模块 alias 必须满足 `applicationAlias.xxx`，创建后不允许手动修改；静态模块可以没有元数据，动态模块必须有一个主元数据。
3. 元数据 `id` 是平台稳定 ID，`metadataAlias` 在应用下唯一，物理表定位使用 `schemaName + tableName`。
4. 模块与元数据通过关系绑定，一个模块最多一个主元数据，关系模型能表达父、子、孙结构。
5. 跨模块体现时，菜单、入口、动作、查询、权限、运行时 URL 和审计口径统一使用当前模块的 `moduleAlias`。
6. 至少跑通一条 `应用 -> 模块 -> 元数据 -> 发布 -> DynamicRecordService.entity(moduleAlias, metadataAlias) -> CRUD/query` 闭环。

### M2 复核结论

M2 的配置态基础链路已经具备进入后续阶段的条件。应用、模块、元数据、菜单和字典都有独立服务契约测试；发布链路已通过真实发布测试覆盖 `应用 -> 模块 -> 主元数据 -> 字段/关系/视图/动作 -> 运行态`。M2 不再继续扩展用户、组织、角色和权限；这些进入 M5 身份与权限闭环。

## M3 动态模块可交付闭环

M3 的目标是让动态模块从基础数据能力推进到运行时交付能力。前半段已经完成字段字典、动态查询、动态引用展示、运行态 descriptor 和基础视图输出，但这只说明动态模块具备“结构可描述”的基础，不等于已经具备完整交付能力。

校准后的 M3 继续聚焦动态平台基础能力，不提前进入业务线。M3 完成时，一个动态模块应至少能被前端或调用方按稳定契约使用：能读取模块结构，能得到列表/表单配置，能解析引用，能看见可执行动作，并且这些配置有后续权限、审计、工作流、导入导出等能力的挂载位置。

M3 不追求完整 UI 设计器，也不在本阶段完成权限、工作流、导入导出、编码规则等应用级能力；这些能力按依赖关系进入后续里程碑。但 M3 必须先把视图、引用、动作和字段行为的基础契约整理清楚，避免后续继续堆临时接口。

| --- | --- | --- | --- | --- | --- |

### M3 复核结论

M3 已满足进入 M4 的条件：动态模块已经具备从配置态发布到运行态、读取结构契约、执行 CRUD/query、解析引用、输出视图/关联视图、治理动作和表达字段行为的基础闭环。模块主元数据已通过 `mainEntityAlias` 显式表达，不再依赖实体列表顺序推导模块动作；真实发布链路已覆盖 descriptor、动作、视图、关联视图和字段行为。当前仍不包含完整权限、工作流、导入导出、编码规则和发布中心治理，这些不再回塞到 M3，而按 M4 以后里程碑逐步展开。

## M4 平台执行底座

M4 的目标不是直接铺开所有应用专题，而是在 M3 动态模块可交付边界稳定后，先补齐后续平台应用能力共同依赖的执行底座。规则、动作、事件、审计、时间字段和 Web/OpenAPI 交付如果没有统一口径，编码规则、导入导出、工作流、推送生成和回写都会被迫各自发明临时机制。

M4 的规则与公式不做简化占位版，而应按可长期承载字段计算、保存校验、条件判断、动作可用性、导入校验和工作流条件的正式引擎推进。确有复杂能力可以分批落地，但模型、执行上下文、错误报告和扩展边界必须按完整闭环设计。

### M4 当前推进口径

M4 已进入执行底座建设，不再回头补 M2/M3 配置交付能力。短期优先级是：先建设动作执行上下文，让规则、动作、操作者、租户和系统态有统一执行边界；随后补事件与审计上下文。动态 Web API 和 OpenAPI 属于交付接口，不应早于动作、规则和审计边界稳定。

## M5 权限与身份闭环

M5 的目标是把身份、角色、动作权限、数据权限和菜单剪枝做成完整闭环。权限不做“只留一个权限字符串字段”的半成品；如果进入实现，就必须能支撑动作执行、动态模块、静态模块、菜单、数据范围和审计上下文。

M5 功能点矩阵在 M4 完成后展开。当前先固定以下边界：

1. 身份上下文至少包括用户、租户、组织、部门、角色和系统态操作者。
2. 动作权限要能同时约束静态模块动作和动态模块动作。
3. 数据权限是独立授权策略，不和 `tenantId` 基线隔离混淆。
4. 菜单权限是菜单树剪枝，不改变菜单本身的租户隔离和方案模型。
5. 权限判定结果要能进入动作执行上下文和审计上下文。

## M6 工作流与任务闭环

M6 的目标是将审批节点和任务节点统一为工作流能力，并形成完整闭环。工作流不做只有状态字段和待办入口的简化版；进入实现后，应覆盖定义、发布、运行、任务、待办、历史、撤回、终止、回退、转办、加签、分支、汇聚和插件挂点等核心能力。

M6 功能点矩阵在 M5 完成后展开。当前先固定以下边界：

1. 工作流节点包括审批节点和任务节点，任务是可独立配置、判定和引导动作的业务对象。
2. 工作流动作必须接入 M4 的动作执行上下文、规则引擎、事件和审计上下文。
3. 工作流权限必须接入 M5 的身份与权限闭环。
4. 工作流状态字段、实例字段和任务字段要兼顾静态模型与动态元数据模型。

## M7 平台业务编排能力

M7 的目标是推进编码规则、导入导出、推送生成和回写等可复用平台应用能力。其中推送生成和回写业务复杂度高，明确后置到本阶段，不提前塞进 M4/M5/M6。

M7 功能点矩阵在 M6 完成后展开。当前先固定以下顺序：

1. 编码规则优先，依赖规则引擎、动作上下文、事件和事务边界。
2. 导入导出其次，依赖 descriptor、字段行为、字典、引用、时间字段和规则校验。
3. 推送生成与回写最后推进，依赖规则、动作权限、系统态写入、事件、审计和必要的工作流挂点。

## M8 业务专题覆盖

M8 的目标是在平台基础能力足够稳定后，再按业务单元推进具体业务覆盖。业务专题不应绕开平台能力直接堆静态代码；如果业务推进时发现平台能力缺口，应优先判断该缺口是否需要回补到公共能力层。

M8 候选专题包括具体业务领域。组织、部门、用户、角色等身份基础能力已前移到 M5，不再作为普通业务专题处理。
