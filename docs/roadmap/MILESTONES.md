# MuYunSpring 里程碑规划

本文用于约束新 MuYun 的阶段推进节奏。当前核心路线是“动静一体”：静态 Java 模型和动态元数据模型不应演化成两套平台，而要共享同一套模型基座、数据访问、能力组合和生命周期语义。

文档只详细展开当前里程碑。后续里程碑只保留阶段目标和进入条件，等前置里程碑完成后再补功能点矩阵，避免过早设计造成文档负担。

## 里程碑总览

| 里程碑 | 名称 | 基本目标 | 进入条件 |
| --- | --- | --- | --- |
| M1 | 动静一体核心底座 | 证明静态模型和动态元数据模型能接入同一套能力体系 | 当前阶段 |
| M2 | 平台标准业务接入 | 将用户、组织、部门、角色等平台内嵌业务按能力组合方式接入 | M1 的静态/动态数据与能力契约稳定 |
| M3 | 动态平台运行时闭环 | 动态模块从能存能查推进到可配置、可运行、可交付 | M2 证明标准业务不游离于平台体系 |
| M4 | 平台应用能力编排 | 补齐工作流、编码规则、导入导出、推送生成、回写等应用级能力 | M3 形成稳定运行时边界 |

## M1 动静一体核心底座

M1 的目标不是先做一批静态业务，也不是先做完整动态平台，而是先证明两类模型能共享一套平台底座。

### 验收目标

1. 静态模型通过 Java 类、注解、DAO 和 Ability 组合获得标准平台能力。
2. 动态模型通过元数据定义实体、字段和表结构，并复用同一套能力语义。
3. 静态 DAO 和动态 DAO 在数据操作层共享一致的查询、分页、排序、软删、版本和生命周期规则。
4. 静态模型自动建表与运行态元数据建表都具备最小闭环。
5. 至少一个静态样例和一个动态样例能证明 CRUD、树、排序、引用等基础能力语义一致。

### M1 当前剩余缺口

1. 真实数据库行为测试已覆盖静态仓储入口、动态 CRUD、树、排序、引用、父子聚合、租户隔离和租户内唯一；后续新增能力仍需要同步补 contract。
2. 租户上下文来源尚未接入身份体系，当前由 `TenantContext` 提供平台基础 scope。
3. 缓存实现仍是进程级静态 Map，适合 M1 验证，不作为最终缓存治理形态。
4. 父子聚合 hook 纪律、动态 marker interface、`enabled` 字段边界等问题已记录到技术债。

### 推进原则

1. 按能力单元推进，而不是按零散小点推进。一个能力单元应尽量包含模型、接口、默认实现、测试和必要的真实数据库验证。
2. 代码、测试、注释、文档各司其职。能由测试表达的细节，不写成长篇设计文档。
3. 平台基础设施允许多走一步。涉及数据操作、能力组合、生命周期、建表和元数据边界时，应优先考虑中长期复用。
4. 不为了拆分而拆分。包和模块边界只有在降低心智负担、减少重复、稳定依赖方向时才继续拆。
5. M1 期间不展开菜单、权限、登录、工作流等平台应用能力，除非它们暴露出必须提前下沉的底层契约。

## M2 平台标准业务接入

M2 的目标是让平台内嵌业务不游离于能力体系之外。用户、组织、部门、角色等业务可以有手写规则，但基础 CRUD、树、排序、引用、生命周期和数据操作不应重复实现。

M2 的功能点矩阵在 M1 完成后再设计。

## M3 动态平台运行时闭环

M3 的目标是让动态模块从基础数据能力推进到运行时交付能力，包括动态字段、动态查询、动态引用、视图、动作、运行时 API 等。

M3 的功能点矩阵在 M2 完成后再设计。

## M4 平台应用能力编排

M4 的目标是补齐更高层的平台应用能力，包括工作流、编码规则、导入导出、推送生成、回写、审计和治理能力。

M4 的功能点矩阵在 M3 完成后再设计。
