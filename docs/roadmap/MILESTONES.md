# MuYunSpring 里程碑规划

本文用于约束新 MuYun 的阶段推进节奏。当前核心路线是“动静一体”：静态 Java 模型和动态元数据模型不应演化成两套平台，而要共享同一套模型基座、数据访问、能力组合和生命周期语义。

文档详细展开已完成的 M1、M1.5 和当前推进的 M2。后续里程碑只保留阶段目标和进入条件，等前置里程碑完成后再补功能点矩阵，避免过早设计造成文档负担。

## M1 动静一体核心底座

M1 的目标不是先做一批静态业务，也不是先做完整动态平台，而是先证明两类模型能共享一套平台底座。

### 验收目标

1. 静态模型通过 Java 类、注解、DAO 和 Ability 组合获得标准平台能力。
2. 动态模型通过元数据定义实体、字段和表结构，并复用同一套能力语义。
3. 静态 DAO 和动态 DAO 在数据操作层共享一致的查询、分页、排序、软删、版本和生命周期规则。
4. 静态模型自动建表与运行态元数据建表都具备最小闭环。
5. 至少一个静态样例和一个动态样例能证明 CRUD、树、排序、引用等基础能力语义一致。

### M1 收尾结论

1. M1 的必要验收已完成：静态模型和动态元数据模型已经能接入同一套基础能力、数据访问、生命周期、租户、软删、版本、树、排序、引用、启停、缓存和父子聚合语义。
2. 真实数据库行为测试已覆盖静态仓储入口、动态 CRUD、动态稳定门面、树、排序、引用标题、引用投影、父子聚合、租户隔离、租户内唯一和复杂 Criteria 组合。
3. M1 不再保留阻塞项。跨节点缓存治理、身份体系租户来源、动态 schema 多实体失败补偿、显式 HYDRATED 深度读取等进入 M1.5 或后续阶段。
4. 后续每补一个能力或门面，仍必须同步检查静态/动态两侧，避免重新分裂成两套平台。

### 推进原则

1. 按能力单元推进，而不是按零散小点推进。一个能力单元应尽量包含模型、接口、默认实现、测试和必要的真实数据库验证。
2. 代码、测试、注释、文档各司其职。能由测试表达的细节，不写成长篇设计文档。
3. 平台基础设施允许多走一步。涉及数据操作、能力组合、生命周期、建表和元数据边界时，应优先考虑中长期复用。
4. 不为了拆分而拆分。包和模块边界只有在降低心智负担、减少重复、稳定依赖方向时才继续拆。
5. M1 和 M1.5 期间不展开菜单、权限、登录、工作流等平台应用能力，除非它们暴露出必须提前下沉的底层契约。

## M1.5 能力层夯实

M1.5 的目标是在进入平台标准业务前，把基础能力从“主链路可用”推进到“边界清晰、默认好用、能支撑中期业务接入”。这一阶段可以吸收成熟平台的优秀做法，但产物必须沉淀为 MuYun 自己的命名、模型和契约。

### 验收目标

1. 生命周期拆出平台内部链和业务扩展链，平台能力不依赖业务 hook 手动调用 `super` 才能正确执行。已在 CRUD 与父子聚合链路完成第一轮收口。
2. 引用能力从标题读取升级为统一引用定义、投影、默认 RAW 加载边界和跨引用缓存失效基础。
3. 缓存能力具备 namespace 生命周期、容量或过期边界、动态运行态发布/卸载清理入口。已完成本地默认边界，后续只在需要监控、跨节点一致性或业务级策略时升级为缓存管理器。
4. 静态和动态写入形成统一乐观锁 CAS contract，旧版本更新和删除必须有明确失败语义。能力层、动态 DAO 和静态 Repository 已统一走底层条件写入口；静态删除走条件删除，动态软删走条件更新。
5. 动态 schema 演进覆盖字段新增、约束变更、字段删除、dry-run 和单实体失败不污染运行态；多实体部分执行补偿进入发布中心设计。
6. Criteria 真实数据库 contract 已覆盖复杂组合、分页排序、非法字段拒绝和 raw 条件边界；后续按新查询能力继续增量扩展。
7. 技术债已按 M2 配置自举、M3 动态运行时、场景触发和运维治理分层；M1.5 基础能力没有必须阻塞 M2 的技术债。
8. 静态 service 和动态门面已经对齐基础查询入口，业务侧可直接通过能力层 `list`、`pageQuery`、`count` 获取统一租户和软删 scope。
9. 每个能力补强都同时检查静态模型和动态模型，避免重新分裂成两套平台。

M1.5 完成前，不以扩展业务线数量作为主要进度指标。

## M2 平台配置与元数据自举

M2 的目标是让平台先能管理自身配置态。第一批聚焦应用、模块、元数据、菜单和数据字典；组织、部门、用户、角色顺延到后续里程碑。M2 不追求完整管理后台 UI，而是证明配置业务能复用能力体系，并能从持久化配置发布到动态运行态。

### 验收目标

1. 应用、模块、元数据、菜单、数据字典使用统一 alias 规则，模块身份统一使用 `moduleAlias`，不引入 `moduleId` 同义词。
2. 模块 alias 必须满足 `applicationAlias.xxx`，创建后不允许手动修改；静态模块可以没有元数据，动态模块必须有一个主元数据。
3. 元数据 `id` 是平台稳定 ID，`metadataAlias` 在应用下唯一，物理表定位使用 `schemaName + tableName`。
4. 模块与元数据通过关系绑定，一个模块最多一个主元数据，关系模型能表达父、子、孙结构。
5. 跨模块体现时，菜单、入口、动作、查询、权限、运行时 URL 和审计口径统一使用当前模块的 `moduleAlias`。
6. 至少跑通一条 `应用 -> 模块 -> 元数据 -> 发布 -> DynamicRecordService.entity(moduleAlias, metadataAlias) -> CRUD/query` 闭环。

## M3 动态平台运行时闭环

M3 的目标是让动态模块从基础数据能力推进到运行时交付能力，包括动态字段、动态查询、动态引用、视图、动作、运行时 API 等。

M3 的功能点矩阵在 M2 完成后再设计。

## M4 平台应用能力编排

M4 的目标是补齐更高层的平台应用能力，包括工作流、编码规则、导入导出、推送生成、回写、审计和治理能力。

M4 的功能点矩阵在 M3 完成后再设计。
